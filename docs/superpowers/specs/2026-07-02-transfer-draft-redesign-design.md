# 转会选秀重构设计文档(手动逐队逐类别 + 挖角)

> 状态:已与用户在 brainstorming 中逐节确认(2026-07-02)。本文件为实现前的权威设计。
> 关系:**取代** [2026-06-25-transfer-market-design.md](2026-06-25-transfer-market-design.md) 第 3 章「转会流程」与第 4.1「推荐选车规则」。
> 预算/薪资公式、合同即状态、每类别 0/2、品牌锁、退役车不入市等**保持不变**,仍以 2026-06-25 及「签约状态合并」文档为准。

## 1. 背景与动机
现有实现(`market.open_market` + `market.recommend`)是**全自动**的:开盘一刀切释放全部短期车 → 算法按余额自动把各队补满。用户实测后要求改为**半自动、逐队逐类别的手动取舍**流程,并引入**从其他车队挖角短期车**的机制。全自动补位降级为**仅供参考的建议**。

## 2. 名词
- **分级车队 / 类别(category)**:一支 `Team` 在 F1 / GT3 / 公路 三个赛事类别下各有一支「分级车队」,各自最多 2 个现役。转会以「(队, 类别)」为最小处理单元。
- **当前队(current team)**:此刻正在页面上被调整的队,同一时刻只有一支。
- **锁定/确认队(locked team)**:已完成调整并定档的队;其车(含短期)退出所有候选池,受保护、本轮不再变动。
- **余额(headroom)** = `budget(参照赛季) − 现役薪资之和`。
- **候选池(pool)**:某 (队, 类别) 一个自由槽位可选的车集合(见 §5)。

## 3. 总体流程
```
开盘(建 MarketDraft + 快照全车 team_id/status)
  └─ 循环,直到没有未处理队:
       重算各未处理队余额 → 取余额最高者 = 当前队
       进入当前队页 → 自动解约该队全部短期车(空出薪资)
       按「含长期车类别优先」的约束,逐个类别:
           看建议(保持不变/完全解散/补强) → 两下拉手动选 → 锁定该类别
       三个类别全锁 → 确认该队 → 加入锁定集合
  └─ 全部处理完(或手动结束) → 开启下赛季
随时可「一键重置本次转会」→ 用快照还原所有车,回到开盘前
```

### 3.1 优先级(外层循环)
- 参照赛季 = 最近一个已结束赛季(`market.reference_season`)。
- 队列按**当前余额从高到低**排序。余额计算时,**未处理队一律按「维持现有阵容」**(短期车按留任计薪)——即优先级反映「假设谁都不动时谁的闲钱最多」。
- **平局裁决**(余额相同时依次):① **车队积分(参照赛季)大者优先**;② 仍相同 → **持有当前 MMR 最高的车的车队优先**(该队现役车中最高赛季末 MMR 与其他并列队比较);③ 仍相同 → **随机**(用当前 `MarketDraft` 内固定的随机种子,保证同一次转会里排序稳定、可复现)。
- **每确认一支队后重算**队列(挖角会降低被挖队承担薪资、抬高其余额,使其自然上移)。已确认队不再入列。

### 3.2 进入当前队:自动解约自家短期(修订 A)
- 当某队成为当前队、进入其选择页时,系统**立即解约该队自己的全部短期车**(短期 → 未签约、`team_id=None`),薪资全部空出。该队基线变为「仅长期车 + 满预算」。
- 被解约的短期车进入候选池;当前队可再签回(「保持不变」建议即把它们重新填入下拉),未签回的留在池中供后续队挑。
- **只解约当前队自己的短期**;其他未处理队的短期车仍留任,但可被当前队挖角(§5)。
- 该自动解约同样记入重置快照,可被「重置」还原。

### 3.3 类别处理顺序:含长期车类别优先(修订 B)
- 一支队内,**所有含长期车的类别必须全部先锁定**,之后才允许锁定不含长期车的类别。
- 目的:长期车(硬支出)及其第 2 个槽位**优先占用预算**,杜绝「先把预算花在无长期车的类别、逼得某个只有 1 辆长期车的类别没钱补强」。
- 因此**预算超支的唯一合法情形**纯粹是「某含长期车类别的长期薪资本身已超预算」,而非被别处花钱挤出来的。
- 校验:存在未锁定的「含长期车类别」时,锁定任何「无长期车类别」的请求被拒。

## 4. 锁定某类别的校验(硬性)
锁定 (队, 类别) 时,依次校验:
1. **顺序**(§3.3):若本类别无长期车、而该队仍有未锁的含长期车类别 → 拒绝。
2. **品牌锁**:厂商队每个非空槽的车必须同品牌或品牌「无」;签后品牌「无」不变。独立队不限。
3. **长期车钉死**:该类别现有长期车占据槽位且**不可移除**(本队或他队皆不可释放长期车)。
4. **现役数量**:
   - **含长期车类别**:须补满到 **2**;**例外** —— 候选池中没有「合法(品牌)+ 买得起(≤ 当前余额)」的第 2 辆车时,允许停在 **1**(长期车单撑)。
   - **无长期车类别**:**0 恒合法**(整类别不参赛);**停在 1** 仅当**预算买不起「两辆最便宜的车」**(两最便宜薪资之和 > 当前余额)或候选池不足 2 辆时才允许;若够买两辆最便宜的,则必须 **0 或 2**(不能只签 1 把钱留着)。
5. **预算**:锁定后该队现役薪资之和不得超预算;**唯一例外** = 校验 4 里「含长期车类别的长期薪资本身超预算」。
6. **来源合法**(§5):每个非空槽的车必须来自候选池;若该车原属其他未确认队的短期合约,则锁定即把其 `team_id` 改到当前队(默认签为短期)。

锁定成功后:即时改写相关 `Car` 行(签约/挖角/释放)、余额刷新、面板进入「已锁定,可解锁重选」态。

> **防死锁不变量**:逃生阀保证**任何单个类别永远至少有一个合法锁定态** —— 含长期车类别可停在 1(无可签车时);无长期车类别 **0 恒合法**(且买得起 1 辆时可停 1)。每次锁定按「此刻」余额判定,单步自足;队列每确认一队严格 −1。故「确认该队」闸门必可达,不会卡死。**取舍**(两个长期类别争预算、先锁者吃预算、后锁者停在 1)由玩家排序决定,系统接受(方案 A)。

## 5. 候选池(某类别一个自由槽)
候选池 = 以下三类之并,再经品牌锁与「≤ 当前余额」过滤(不满足预算的车在下拉中标注、不可选中):
- 本队该类别现有车(含刚被自动解约、现为自由身的自家原短期车);
- 全局**自由身**(未签约)该类别车;
- **尚未确认队**的该类别**短期**车(= 挖角对象)。
排除:长期车、已确认队的任何车、退役车。

## 6. 建议(仅供参考,可无视)
每个 (队, 类别) 面板顶部给三种一键建议,点击后自动填入两下拉,仍可手改:
- **保持不变**:填入该队进队前的原阵容(自家原短期车重新填回)。
- **完全解散**:两槽全空 —— **仅当该类别无长期车**时可点。
- **补强**:在「品牌 + 买得起 + 候选池可得」范围内,选使**两槽合计赛季末 MMR 最高**的组合;若有长期车钉死一槽,则为另一槽选可得的最高 MMR 车。**建议永远是合法锁定态**:无长期车类别够组 2 就给 2(高 MMR 贪心没凑到 2 时退取两辆最便宜的),否则给可行的 1 或 0。
`market.recommend` 的旧全自动补位算法**降级/改写**为本节「补强」的单类别计算 `category_recommendation`。

## 7. 数据模型
新增(经 `app/db.py::sync_schema` 自动迁移):
- `MarketDraft`:`id`、`reference_season_id`、`current_team_id: Optional[int]`、`locked_team_ids: str`(如逗号分隔或 JSON)、`tiebreak_seed: int`(开盘时定,供 §3.1 平局③随机裁决,保证排序稳定可复现)、`created_at`。每次开盘一条;定档/开启下赛季后关闭或删除。
- `DraftCarSnapshot`:`id`、`draft_id`、`car_id`、`orig_team_id: Optional[int]`、`orig_status`。开盘时对**全部车**各存一行,供「一键重置」还原。

> 复用现有未被读取的 `TeamBudget/CarSalary` 快照表**不**纳入本设计(它们仍按现状,或按 P2 结论清理,与本重构解耦)。

## 8. 服务与端点
`app/services/market.py`(重写流程编排,保留 `reference_season/season_pair/committed_salary/headroom/compute*`):
- `open_market(session)`:建 `MarketDraft` + 写 `DraftCarSnapshot`;**不再**一刀切释放。
- `draft_queue(session)`:返回未处理队按余额排序 + 各队锁定态 + 当前队。
- `enter_team(session, team_id)`:设为当前队并自动解约其短期(§3.2);校验它确为队列最高余额者(§10:严格取最高,不允许任意跳选)。
- `category_pool(session, team_id, category)`:§5 候选池(带薪资、可负担标记)。
- `category_recommendation(session, team_id, category)`:§6 建议三选项的具体车。
- `lock_category(session, team_id, category, slot1_car_id, slot2_car_id)`:§4 校验 + 应用(release/sign/poach);记录该类别已锁。
- `unlock_category(session, team_id, category)`:回退该类别到进队后的初始态。
- `confirm_team(session, team_id)`:要求 3 类别全锁 → 加入 `locked_team_ids` → 清 `current_team_id`。
- `reset_draft(session)`:用 `DraftCarSnapshot` 还原所有车,清空锁定集合与当前队。
- `finalize(session)`:结束转会、开启下赛季(沿用现有开季钩子),关闭 draft。
- 删除:旧 `open_market` 一刀切释放、旧全自动 `recommend`。

`app/routers/market.py` + 模板(HTMX 局部刷新):
- 队列区(§3.1):余额排序、当前队高亮、已确认打勾、「重置」「结束转会/开启下赛季」。
- 当前队区:F1/GT3/公路三面板;含长期车类别在未锁完前禁用无长期车面板;每面板:建议按钮 + 两下拉(长期车钉死禁用槽)+「锁定/解锁」;余额实时条。
- 「确认该车队」按钮:三类别全锁后亮起。

## 9. 测试要点
- 优先级:初始按「维持阵容」余额排序;确认一队后重算;挖角抬高被挖队余额。
- 平局裁决:余额相同→车队积分大者优先;再相同→持最高 MMR 车的队优先;再相同→按固定种子随机且稳定。
- 进队自动解约自家短期;他队短期不被动解约。
- 顺序约束:未锁完含长期车类别时,拒绝锁无长期车类别。
- 锁定校验:品牌锁、长期钉死不可移除、含长期类别补满到 2(逃生阀允许 1)、无长期类别 0/2(**买不起两辆最便宜的才允许 1**)、预算(仅长期合法超支)、挖角改归属。
- 逃生阀/防死锁:负余额队、厂商队无合法车、池空、无长期类别只 1 辆可行等情形均能锁定并确认;队列必然收敛。
- 建议三选项返回正确车且**永远可直接锁定**;补强取 MMR 最高可负担组合(无长期类别够组 2 必给 2、否则给可行的 1 或 0)。
- 重置:快照还原所有车与集合。
- finalize:定档后可开启下赛季;路由渲染与各端点校验。

## 10. 边界与约定(已定)
- 一时刻仅一支当前队;进队即自动解约其短期,故「进队」不可轻率——但有「重置」兜底。
- **当前队严格取队列最高余额者**(不让用户任意跳选),以贴合「余额最高优先重建」的设计。
- 无长期车的类别:**0 恒合法**;凑不齐 2(买不起两辆最便宜的 / 候选不足 2)时允许停在 1;够买两辆最便宜的则必须 0 或 2。
- 退役车:全程不入市、不计薪资、不占名额、留原队(不受自动解约影响)。
- 首赛季无往季数据的处理沿用 2026-06-25 文档 §5。

## 11. 暂不做(YAGNI)
- 长期车的主动解约/转会费/违约金。
- 多人/并发编辑(本地单用户,顺序操作)。
- 预留预算的方案 B(§4 采方案 A,由玩家取舍)。
